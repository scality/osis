# Deltas from the Confluence source doc

Tracked while running the lab on Rocky 9 (doc was written for Rocky 8).

| # | Doc says | Reality | Resolution |
|---|---|---|---|
| 1 | `dnf module enable postgresql:13` | Rocky 9 ships only `postgresql:15` and `postgresql:16`. PG 13 reached EOL 2025-11. | Use `postgresql:15` (VCD 10.5 documents PG 13–15 compat). |
| 2 | OSE RPM filename in doc: `vmware-ose-2.2.2-22098306.el7.x86_64.rpm` | We have `vmware-ose-3.0.0-23443325.el8.x86_64.rpm` | Use the newer 3.0 / el8 build; expect command surface differences vs the doc's OSE 2.x examples. To verify during install. |
| 3 | Doc instructs `sudo su -` then disconnect and SSH as root | We stay in tmux and use `sudo -i` once; no SSH reconnect needed | The `sed -i 's/^.*ssh-rsa/ssh-rsa/' authorized_keys` step is still useful for future direct-root SSH but not required for the install flow. |
| 4 | Doc shows firewall step "should be disabled, but if not..." | Rocky 9 AMI has no firewalld unit | `systemctl stop firewalld` returns non-zero; script must tolerate. |
| 5 | n/a (capture tooling) | `tmux send-keys` mangles SQL single quotes (escapes are interpreted at the wrong layer) | Drop multi-line scripts to `/tmp/*.sh` via SSH `cat <<'EOS'`, then `bash /tmp/*.sh`. Codify in install scripts. |
| 6 | Doc: `service vmware-vcd start` works after installer creates `/etc/init.d/` symlink | Installer fails to create the symlink on Rocky 9 (`/etc/init.d/` missing) and skips the systemd unit because `/sbin/chkconfig` is gone | Install script must (a) `dnf install -y initscripts` so `/etc/init.d/functions` exists and the VCD init script can be sourced; (b) write `/etc/systemd/system/vmware-vcd.service` ourselves (forking type, ExecStart=`/opt/vmware/vcloud-director/bin/vmware-vcd start`); (c) `systemctl daemon-reload && systemctl enable --now vmware-vcd`. |
| 7 | Doc shows `/opt/vmware/vcloud-director/bin/configure` succeeding cleanly | Same as #6: `configure` prints `/sbin/chkconfig: No such file or directory` at line 105 but exits 0 — DB schema is initialized correctly | Non-fatal; install script can tolerate. |
| 8 | Doc VCD version: 10.5.1-23401219 | Same installer, but the RPMs inside are now `vmware-vcloud-director-33.2024.02.29-23400185` | Cosmetic; same .bin works. |
| 9 | Doc implies the H5 UI works immediately after `system-setup` | `https://<IP>/provider` returns "Failed to Start" because (a) the `sites` table has empty `rest_api_endpoint`, `base_ui_endpoint`, `multisite_url`, `name`, and (b) accessing via the raw IP still fails even after populating those — H5 UI requires the FQDN. The VCD cert SAN contains the FQDN (`ip-<a>-<b>-<c>-<d>.<region>.compute.internal`), the short hostname, and the IP, but the H5 UI only initializes cleanly when the URL matches the FQDN form. | (1) `UPDATE sites SET rest_api_endpoint=, base_ui_endpoint=, multisite_url= to https://<FQDN>, name='VCD' WHERE is_local_site=true;`. (2) `systemctl restart vmware-vcd`. (3) On the client Mac: add `<IP> <FQDN>` to `/etc/hosts` and browse to `https://<FQDN>/provider`. Trust the VCD cert in the Mac keychain. |
| 10 | Doc: `openssl pkcs12 -export ... -password pass:passwd` for the PKCS#12 build | OpenSSL `-password` only sets the *output* PKCS#12 password. The *input* cert.key was also encrypted (with the same passphrase) so OpenSSL prompts interactively. | Pass both: `-passin pass:passwd -password pass:passwd`. |
| 11 | Doc (OSE 2.x): `ose -h` prompts for EULA once; `ose director set --user admin@system` accepts password via stdin | OSE 3.0 prompts EULA on the **first** invocation (then persists to `/opt/vmware/voss/agreement`) AND `director set` additionally prompts "Do you trust this certificate for the SSL connection? (y/N)" with the VCD self-signed cert fingerprint. Doc was written before this extra prompt was added. | First-time invocation: pipe `accept\n`. For `director set`, pipe `<password>\ny\n` (password + trust-cert response). `ose cert import` puts the cert in OSE's *server* keystore, not the outbound truststore — it doesn't make OSE trust VCD; the y/N prompt does. |
| 12 | n/a (capture tooling) | OSE emits ANSI cursor-position queries on every command; replies (`^[[18;162R`) leak into stdin and pollute the next prompt. | Send `reset` periodically, or wrap ose calls with `script -q` / `setterm -clear` / disable terminal reporting. Investigate during install-script work. |
| 13 | Doc: `ose endpoint set --region=us-east-1 --url=https://<IP of VM>:8443` (the doc uses the IP) | VCD H5 UI's Object Storage panel makes the OSE API calls directly **from the browser**. If the URL is an IP and the cert is trusted by FQDN (because we set up VCD over FQDN), the browser shows "Object Storage Extension service endpoint is found not accessible". The cert SAN does include the IP, but the Mac keychain `add-trusted-cert` of the VCD cert doesn't propagate IP-based trust well across browsers. | Use FQDN for both: re-import OSE cert with `-name <FQDN>` (so the keystore alias matches the cert CN), and `ose endpoint set --url=https://<FQDN>:8443`. Restart OSE. Ensure `/etc/hosts` on the client Mac maps the FQDN to the VM's private IP. |
| 14 | Doc has no mention of CORS | After fixing the OSE-cert hostname mismatch, the Object Storage panel still fails with `CORS error` on the `current-user` XHR. OSE allows CORS only for the registered Cloud Director URL. If VCD is accessed at the FQDN but OSE has `https://<IP>` as its director URL, every XHR from VCD's UI is cross-origin. | After switching VCD to FQDN, also re-run `ose director set --url https://<FQDN> --user admin@system` so OSE's allowed-origin matches the actual VCD origin, then `ose service restart`. |
